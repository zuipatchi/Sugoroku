# sync-docs

ドキュメントとコードの乖離を検出・修正するスキル。

## フロー

1. **調査** — Explore エージェントにドキュメントとコードを横断的に照合させる
2. **修正** — 乖離箇所を Edit ツールで修正する
3. **コミット** — 変更をコミットする

---

## Phase 1: 調査

Explore エージェントを spawn して以下を調査させる。エージェントへの指示：

- `docs/` 配下の全 .md ファイルを読む
- `CLAUDE.md` を読む
- CLAUDE.md の「主要ファイルの場所」テーブルに列挙されたパスが実際に存在するか確認する
- 各ドキュメントが言及しているクラス・メソッド・プロパティ・定数が実際のコードに存在するか確認する
- ゲームルールや計算式が記述されている場合、実装と一致しているか確認する
- アーキテクチャ・DI 登録・フロー説明がコードと一致しているか確認する

乖離の候補が見つかったら、**ドキュメントの該当行** と **実コードの実態** を両方報告させる。

---

## Phase 2: 乖離の分類と修正

エージェントの報告を受けて、乖離を以下の 3 種類に分類する：

| 種別 | 判断基準 | 対応 |
|---|---|---|
| ドキュメントが古い | コードが正しい実装で、ドキュメントが旧仕様を指している | ドキュメントを修正 |
| ドキュメントが正しい | コードが未実装または誤実装 | ユーザーに確認してから判断 |
| 双方が曖昧 | どちらが正しいか不明 | ユーザーに確認 |

コードとドキュメントのどちらが「正」かが明らかでないケースは、**自己判断で修正せず** ユーザーに確認する。

修正時の注意：
- 削除した API・定数があれば記述ごと削除する（「削除済み」コメントを残さない）
- 新しく追加された API があれば漏れなく追記する
- 文体・スタイルは既存ドキュメントに合わせる

---

## Phase 3: コミット

修正が完了したらコミットする。複数ファイルを修正した場合も 1 コミットにまとめてよい。

コミットメッセージの形式：
```
ドキュメント修正（<変更の要点>）
```

例：
```
ドキュメント修正（DeckModel API・デッキ構築ルール・CPUデッキの乖離を解消）
```

---

## 注意事項

- コードを読まずにドキュメントだけを書き換えない。必ず実装を確認してから修正する
- ファイルパスの確認は Glob ツールで行う（存在しないパスを「存在する」と書かない）
- メソッド・プロパティの確認は Grep ツールで行う
- 修正範囲は乖離箇所のみ。無関係なリファクタや文言改善は行わない

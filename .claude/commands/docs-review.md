# docs-review

ドキュメントの**内部品質**を検出・修正するスキル。コードは見ず、ドキュメント同士・ドキュメント内の整合性に責務を絞る。

> **sync-docs との違い**: sync-docs は「コードとドキュメントの乖離」を扱う。docs-review は「ドキュメント内部の品質」（矛盾・重複・リンク切れ・誤字・構成）を扱う。コードと突き合わせたいなら sync-docs を使う。

## フロー

1. **調査** — Explore エージェントにドキュメントを横断的にレビューさせる
2. **報告と確認** — 検出した問題をカテゴリ別に一覧表示し、どれを直すかユーザーに確認する
3. **修正** — 承認された箇所を修正する
4. **コミット** — 変更をコミットする

---

## Phase 1: 調査

Explore エージェントを spawn して以下を調査させる。**対象は `docs/` 配下の全 .md・`CLAUDE.md`・`README.md`。コードは読まない**。

`CLAUDE.md`・`README.md`・`docs/` は役割が近く内容が重複・矛盾しやすいので、**3者をまたいだ横断チェックを必ず行う**（例: 同じアーキテクチャ説明が CLAUDE.md と docs/architecture.md の両方にある、README とドキュメントでセットアップ手順が食い違う、CLAUDE.md の「主要ファイルの場所」と docs/ のリンク先がズレている等）。

エージェントへの指示（検出カテゴリ）：

| カテゴリ | 検出内容 |
|---|---|
| 矛盾 | 同じ事項について別々のドキュメント（または同一ドキュメント内）で食い違う記述がないか（数値・命名・手順・方針など） |
| 重複 | 同じ説明・同じ表・同じコード例が複数箇所に重複していないか。一方に集約して他方からリンクすべき箇所はどこか |
| リンク切れ | `[テキスト](パス)` のリンク先ファイル・アンカーが実在するか（相対パス・`#見出し` アンカー含む） |
| 古い記述 | 「TODO」「未実装」「予定」など、もう実態と合っていなさそうな記述。日付・バージョン番号の古さ |
| 構成の崩れ | 見出し階層の飛び（h2 の次にいきなり h4 等）、目次と実際の見出しのズレ、表のカラム不揃い |
| 誤字・表記ゆれ | 明らかな誤字、同一用語の表記ゆれ（例: "Addressable" と "Addressables"、全角/半角の不統一） |

各問題について **「ファイルパスと行番号」「カテゴリ」「問題の要約」「修正案」「確実さ（明確／要判断）」** を報告させる。

> リンク切れの確認は Glob ツールで実ファイルの存在を、Grep ツールで見出しアンカーの存在を確かめる（推測で「切れている」と書かない）。

---

## Phase 2: 報告と確認

エージェントの報告を受けて、問題をカテゴリ別・確実さ別に一覧表示し、どれを修正するかユーザーに確認する。

- **明確**（誤字・リンク切れ・単純な矛盾）はまとめて修正提案する
- **要判断**（重複の集約方針・どちらの記述を「正」とするか・構成の組み替え）は選択肢を示して議論してから決める
  - 特に「矛盾」はどちらが正しいかドキュメントだけでは決められないことがある。その場合は**自己判断で片方に寄せず**ユーザーに確認する
- ユーザーが「やめる」と言ったらコミットせず終了する

---

## Phase 3: 修正

承認された箇所を Edit ツールで修正する。

修正の方針：
- **重複の集約** → 最も適切な 1 箇所に内容を残し、他方は要約＋該当箇所へのリンクに置き換える（情報を二重管理しない）
- **リンク切れ** → 正しいパス・アンカーに直す。リンク先が消えていれば記述ごと見直す
- **矛盾** → 「正」と確定した側に統一する
- **誤字・表記ゆれ** → 既存ドキュメントで多数派の表記に合わせる

注意：
- 文体・スタイルは既存ドキュメントに合わせる
- 修正範囲は検出した問題のみ。無関係な文言改善・リライトはしない
- 内容を消すときは「削除済み」コメントを残さず記述ごと消す

---

## Phase 4: コミット

修正が完了したらコミットする。複数ファイルを修正した場合も 1 コミットにまとめてよい。

コミットメッセージ形式：
```
docs: <変更の要点>
```

例：
```
docs: ドキュメント間の重複を集約しリンク切れ・表記ゆれを修正
```

---

## 注意事項

- このスキルは**ドキュメントの内部品質だけ**を見る。コードとの整合性は確認しない（それは sync-docs の責務）
- リンク・アンカーの実在は必ずツールで確認する。推測で判断しない
- 重複の集約や構成の組み替えは判断を伴うため、必ずユーザー確認を経てから実施する
